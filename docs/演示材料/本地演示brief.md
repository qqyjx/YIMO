## 二、功能演示（15 分钟，按此顺序点击）

### 1. 首页（2 分钟） — 建立整体印象

**打开** `http://localhost:5000/`

**要讲**：
- 左侧导航 = 对象管理平台核心功能，共 12 个面板
- 右上角业务域下拉（22 个域，已接入输配电 / 计划财务 / 计财 / shupeidian 四个抽取结果）
- 首页卡片：对象总数 **16**、关联总数 **13,652**、EAV 实体 **23.5 万**、EAV 值 **293 万**

**要答的问题**：
- Q "这些对象从哪来？" → 对象是从 BA/DA/AA 三份 Excel 中用 SBERT 向量聚类抽出来的高层概念（项目/设备/资产...），有跑一遍算法就出这些对象
- Q "为什么 16 个不是更多？" → 聚类粒度自适应 `max(5, min(20, 实体数/20))`，并对只有 1-3 个实体的垃圾簇做合并，输出的都是有规模的抽象对象

### 2. 对象抽取 + 三层关联（4 分钟） — **演示重点**

**对应甲方核心需求**：对象 ↔ 概念实体/逻辑实体/物理实体的三层关联可视化。

**点击路径**：左侧 "对象管理" → 随便选一个对象比如 "项目 (OBJ_PROJECT)" → 右侧关联面板

**要展示**：
- 三列布局：**概念实体 (紫)** ｜ **逻辑实体 (绿)** ｜ **物理实体 (橙)**
- 项目对象 → 概念层 6 个 / 逻辑层 23 个 / 物理层 23 个关联
- 穿透路径：逻辑实体通过 `via_concept_entity` 桥接到概念实体，让甲方看"层级穿透"落地

**要讲**：
> 这正是甲方 3.1 反馈里要求的"通过业务对象做中间匹配把逻辑/物理层连起来"的实现。一个概念实体可以有多个逻辑实体，一个逻辑实体只归属一个概念实体，我们严格按这个约束建的关联。

### 3. 知识图谱（3 分钟） — **视觉冲击**

**点击路径**：左侧 "三层架构图谱" → 选择对象 → 深度 3

**要展示**：
- ECharts 力导向图，对象为中心，三层实体呈三色辐射
- 桑基图（左侧另一个面板）展示"对象 → 概念 → 逻辑 → 物理" 4 层数据流向

**要讲**：
> 甲方 3.13 提到"建议用知识图谱的展示方式"，这里两种可视化都做了：力导向看关系拓扑，桑基看流向汇聚。

### 4. 全生命周期（2 分钟）

**点击路径**：左侧 "全生命周期管理" → 选对象

**要展示**：
- 4 阶段时间线：立项 → 设计 → 施工 → 运行
- 每阶段展开看到该阶段对象的字段快照
- 16 个对象 × 4 阶段 = 64 条生命周期记录，库里都有

**要讲**：
> 甲方 4.3/4.9 反馈强调"全生命周期管理要展示所有字段"和"统一考虑所有对象"，已落地。

### 5. 穿透式溯源 + 机理函数 + 预警（2 分钟，走马观花）

**点击路径**：左侧依次 "穿透式溯源" → "机理函数" → "预警辅助决策"

**要展示**：
- 溯源链：结算穿透 / 合同审计 / 资产生命周期 三条预置链 + 15 个节点
- 机理函数：3 个预置（阈值/公式/规则），点"测试"可以实时验证输入值
- 预警：按规则扫描 EAV 数据触发预警记录，可"处理"

**要讲**：
> 0.md 愿景需求里的"穿透式监管 + 机理函数 + 预警辅助决策"三件套，这里给出可交互的原型证明可行。具体业务规则库需要甲方后续补齐。

### 6. 治理看板（1 分钟）

**点击路径**："数据治理看板"

**要展示**：完整性卡片 + 跨域对比矩阵 + 缺陷列表

**要讲**：
> 基于 4 个治理 API + 2 个 SQL 视图出的指标，可以一眼看哪些对象的三层关联有缺失。

### 7. 技术说明（1 分钟，收尾）

> 底层用 EAV + SBERT 语义聚类 + DeepSeek LLM 命名，支持多域动态扩展，新加一个域只要把 Excel 放进 `DATA/` 就能识别。

## 三、部署方案介绍（10 分钟）

### 3.1 总体路线：双版本并行

**一页讲清楚**：

```
┌──────────────────┐         ┌──────────────────┐
│  本地演示版       │         │  远程部署版      │
│  webapp/         │         │  platform/       │
│  (过第三方测试)   │         │  (南网服务器上线) │
├──────────────────┤         ├──────────────────┤
│ Python + Flask   │         │ Java + Spring    │
│ MySQL            │         │ 达梦 DM8         │
│ Jinja2 + ECharts │         │ Vue 3 + 南网组件  │
└────────┬─────────┘         └────────┬─────────┘
         │                            │
         └──────── 共享 ─────────────┘
                DATA/ 22 域 Excel
                docs/ 需求与规范
                outputs/ 抽取结果 JSON
```

- **本地演示版**：功能迭代快，甲方评审就看这个
- **远程部署版**：严格按南网三份规范（Java 代码规范、达梦数据库规范、前端代码规范）重写，交付南网研发 V 区

两版**不共享运行态**，只共享数据和需求文档。这样甲方功能评审和合规上线两条线同时推进、互不影响。

### 3.2 远程部署版的合规对齐

严格按 3 份规范（来自 [docs/code/开发规范.zip](../code/开发规范.zip)）：

| 规范 | 落地位置 | 关键约束 |
|------|----------|----------|
| 南网数研院 JAVA 代码规范 | `platform/backend/src/main/java/com/csg/twinfusion/` | 类 UpperCamelCase、方法 lowerCamelCase、Service 前缀 get/list/save/update/remove、包名单数小写 |
| 达梦数据库设计开发规范 | `platform/dm-schema/*.sql` | 表名 `TF_<二级域>_<实体>` 大写下划线；必含 `CREATE_TIME` / `UPDATE_TIME`；每表每字段 COMMENT；布尔 CHAR(1) CHECK |
| 数字平台前端代码规范 | `platform/frontend/` | Vue 3 + TS + 南网组件库（现场取后替换 ElementPlus 占位） |

已实现：
- Spring Boot 3 + 达梦双 profile 骨架（`mvn spring-boot:run` 可直接起 :8080）
- API Phase 1 七端点与 webapp 对齐（health / domains / objects / objects/{code}/relations / stats / stats/domains / summary）
- DM8 DDL 核心 6 表（EAV 4 + OM 2）
- Vue 3 + Vite 前端壳 + 两页示例
- 部署 Dockerfile（backend / frontend）+ docker-compose

代码仓：`github.com/qqyjx/YIMO` 私有仓，全部历史可追溯。

### 3.3 算法接入方案

对象抽取算法的三种接入路线，我们推荐**阶段式落地**：

| 阶段 | 方案 | 做法 | 理由 |
|------|------|------|------|
| 现在 → 上线演示 | C 离线预计算 | 本地跑完抽取 → `outputs/*.json` 带上服务器 | 零算法耦合，最简，当前演示已靠这条路径 |
| 上线演示后 | A Python 微服务 | `object_extractor.py` 打独立容器，Java 后端 HTTP 调 | Python 生态便于算法迭代；Docker 隔离易运维 |
| 南网禁 Python 时 | B JVM 原生 | DJL + ONNX 在 Java 里跑 SBERT | 单语言栈兜底方案 |

**DeepSeek LLM 命名**：若南网内部 LLM 网关可用，改 `DEEPSEEK_API_BASE` 环境变量一项即可切过去；若暂不可用，降级走 `simple_extractor.py` 的关键词规则命名，不影响主功能。

## 五、Q&A 预案（常见问题）

| 问 | 答 |
|----|----|
| 为什么不直接 Java 写？ | Python 原型迭代快，功能验证优先；合规版同步写，两条线各司其职 |
| 南网服务器没网怎么办？ | 所有依赖离线带上（见 4.2），Maven/npm/Docker 都有本地方案 |
| 达梦跟 MySQL 兼容性多大？ | DDL 级差异主要在 IDENTITY / COMMENT / 关键字；我们已写达梦专用 DDL，现场由 DBA 过关键字检查 |
| SBERT 模型占多少资源？ | 400MB 权重，768 维向量；22 域 × 千级实体 CPU 侧几分钟跑完，32C 机器宽裕 |
| 系统能承载多少数据？ | EAV 模型水平扩展，现已 23 万实体 / 293 万值稳定运行；达梦侧按分区建索引可再扩 10× |
| LLM 调用出问题降级吗？ | 有 `simple_extractor.py` 关键词回退路径，不依赖 LLM 也能跑出合理对象 |
| 部署后 12 个月期满怎样？ | 若过验收则续签或迁正式环境；全部代码在南网内网仓可交接 |
