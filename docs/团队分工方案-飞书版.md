# 南方电网智能数据管理项目 — 团队分工方案

> **版本**: v1.0 | **日期**: 2026-03-15 | **适用场景**: 内部汇报

---

## 1. 项目全景

本项目包含**两个子系统**，共同服务于南方电网数字化转型：

| | 系统 1：对象管理器（YIMO） | 系统 2：函数编排服务（低代码平台） |
|---|---|---|
| **定位** | 数据底座 — 业务对象抽取与三层架构关联 | 业务编排 — 函数低代码可视化编排平台 |
| **当前状态** | 核心功能**已完成**，进入维护迭代 | **从零启动**，需完成全部设计与开发 |
| **核心能力** | 自动抽取业务对象 + 三层穿透关联 + 知识图谱可视化 | 机理模型函数封装 + DAG 工作流编排 + 拖拽式画布 |
| **技术栈** | Python Flask + MySQL + SBERT + ECharts | Python Flask + MySQL + Vue3 + AntV X6 |
| **代码量** | **~12,000 行**（已完成） | 待开发 |
| **关系** | 提供业务对象数据 API → | ← 调用对象元数据，编排业务流程 |

## 2. 人员分工（三人制）

| 成员 | 负责系统 | 角色定位 | 精力分配 |
|------|----------|----------|----------|
| **负责人（本人）** | YIMO 对象管理器 | 系统维护 + 甲方对接 + 两系统协调 | YIMO **80%** / 协调 **20%** |
| **成员 A** | 低代码平台 | 后端核心开发（DAG 引擎 + 函数服务） | 低代码 **100%** |
| **成员 B** | 低代码平台 | 前端开发（可视化画布 + 界面） | 低代码 **100%** |

> 注：成员 A/B 的前后端分工可根据实际技术背景调整。如两人均偏后端，改为按模块划分（A 负责 DAG 引擎 + 函数服务，B 负责资产库 + 前端界面）。

### 职责详情

**负责人：**
- YIMO 系统维护与甲方需求响应
- 两系统数据互通接口定义
- 低代码平台关键设计评审
- 验收汇报准备与项目结题协调

**成员 A：**
- DAG 工作流引擎（拓扑排序、节点调度、数据流传递）
- 函数服务 Wrapper（Docker → HTTP 标准化封装）
- 函数资产库后端（CRUD + 分类管理）
- 全链路场景验证与性能优化

**成员 B：**
- 低代码平台前端主界面（Vue3）
- DAG 可视化编排画布（AntV X6：拖拽、连线、参数配置）
- 函数资产库前端（组件库 + 搜索）
- UI 体验优化 + 操作手册编写

## 5. 协作机制

### 沟通节奏

| 频率 | 形式 | 内容 |
|------|------|------|
| 每周一次 | 周会（线上/线下） | 进度汇报 + 阻塞讨论 + 下周计划 |
| 每日 | 群消息 | 关键进展同步 + 紧急问题 |
| 每月一次 | 里程碑评审 | 交付物验收 + 计划调整 |

### 代码协作

| 规范 | 说明 |
|------|------|
| 代码仓库 | Git，每个系统独立仓库 |
| 分支规范 | `feature/<功能名>` / `fix/<问题名>` |
| 提交规范 | Conventional Commits（`feat:` / `fix:` / `docs:` / `chore:`） |
| 代码评审 | 关键模块（DAG 引擎、安全相关）须交叉 Review |

### 两系统接口对接

| 对接项 | 说明 |
|--------|------|
| 共享数据库 | 两系统共用 MySQL，YIMO 业务对象数据可被低代码平台引用 |
| 接口规范 | 负责人在 Phase 1 输出《系统间接口规范》 |
| API 调用 | YIMO 提供业务对象查询 API → 低代码平台调用获取对象元数据 |

| 风险 | 影响 | 应对措施 |
