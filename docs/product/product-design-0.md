# 第一部分：理解 Palantir — 全球最大的数据操作系统公司

## 1. Palantir 是谁

### 1.1 公司背景

Palantir Technologies，中文译名帕兰提尔科技公司，成立于 2003 年，总部位于美国丹佛。公司名字来源于《指环王》中的"真知晶球"，一种可以看到远方事物的魔法水晶球。这个名字精确地概括了 Palantir 的核心使命：帮助组织从海量、零散、混乱的数据中"看见"真相。

Palantir 最初由美国在线支付公司 PayPal 的联合创始人彼得·蒂尔出资创建，第一个客户是美国中央情报局 CIA。CIA 的需求非常明确：情报分析师每天面对来自数十个系统的情报数据，包括卫星照片、通信记录、线人报告和公开新闻，这些数据分散在不同数据库中，格式不统一，无法快速关联分析。Palantir 为 CIA 建造的第一个产品叫 Gotham，名字取自蝙蝠侠的城市。Gotham 解决了一个核心问题：把所有数据源接入一个统一平台，让分析师可以在不同数据之间快速建立关联。

从 2010 年开始，Palantir 将同样的思路带入商业领域，推出了面向企业客户的产品 Foundry。Foundry 的客户包括空客、BP 石油、英国国家医疗服务体系 NHS、日本三井住友银行等。截至 2025 年，Palantir 年收入超过 28 亿美元，市值一度超过 2500 亿美元，是全球估值最高的数据分析公司之一。

要理解 Palantir 的价值，需要先理解一个现实：大多数大型组织并不缺数据，它们缺的是把数据变成行动的能力。一家电力公司可能有 50 多套业务系统，企业资源计划系统 ERP 管财务，制造执行系统 MES 管生产，地理信息系统 GIS 管地图，数据采集与监视控制系统 SCADA 管设备。每套系统都有自己的数据库、自己的表结构、自己的编码规范。当管理层想回答一个简单的问题，比如"我们在广东省的变压器中，哪些运行超过 15 年且近三个月故障率上升了？"，没有人能在一小时内给出答案。因为回答这个问题需要同时查询设备台账系统确认哪些变压器在广东，查询生产管理系统确认运行年限，还要查询检修管理系统获取故障记录，而这三个系统之间没有统一的设备编码。Palantir 要解决的就是这类问题。

### 1.2 两条产品线

Palantir 有两个核心产品，分别面向不同的客户群体。

**Gotham** 面向政府和军事客户。美国国防部、CIA、联邦调查局 FBI、国家安全局 NSA 以及多个北约盟国的情报机构都在使用 Gotham。它的核心能力是将来自不同情报来源的数据整合到一个统一的分析界面中，这些数据涵盖人力情报、信号情报、开源情报和地理空间情报。分析师可以借助 Gotham 发现隐藏的关联模式，比如一个恐怖嫌疑人的通信网络、资金流向和地理轨迹之间的关系。Gotham 的界面类似于一个超级版的"案件分析板"，分析师可以在地图、时间线、关系图之间自由切换。由于涉及国家安全，Gotham 的具体技术细节很少公开。

**Foundry** 面向企业客户，这也是我们重点关注的产品。Foundry 的定位是"数据操作系统"，就像 Windows 是个人电脑的操作系统一样，Foundry 要成为企业数据的操作系统。操作系统的核心价值在于提供一个统一的抽象层：Windows 让你不需要知道硬盘是三星还是西部数据就能存文件，Foundry 让你不需要知道数据在 SAP 还是 Oracle 还是 Excel 里就能使用数据。

## 2. Palantir Foundry 的核心概念

为了帮助读者理解 Foundry 的工作原理，我们先从一个简单的比喻开始，然后逐步深入到技术细节。

### 2.1 Ontology（本体）—— 把数据变成"东西"

**从表格到对象的思维跳跃**

想象你经营一家面包店。你可能有一张 Excel 表格记录每天的销售：

| 日期 | 产品 | 数量 | 单价 | 客户 | 备注 |
|------|------|------|------|------|------|
| 3/15 | 法棍 | 2 | 8元 | 张三 | 老客户 |
| 3/15 | 吐司 | 1 | 12元 | 李四 | 新会员 |
| 3/15 | 法棍 | 3 | 8元 | 张三 | — |

在表格思维下，这只是一行行记录。但如果我们换一种方式思考，"法棍"不是一行数据，而是一个**东西**。这个东西有自己的属性，比如单价 8 元、保质期 1 天、原料是面粉和酵母。它有自己的行为，卖出一个就减少库存。它还和其他东西有关系，法棍的原料来自"面粉供应商 A"，买法棍最多的客户是"张三"。

这就是 Palantir 所说的 Ontology（本体）的核心思想：**不要把数据当成表格里的行，而是把它当成真实世界中的"东西"**。在 Foundry 的术语中：

- **Object Type，对象类型**，相当于模板。比如"面包"是一个对象类型，定义了面包应该有名称、单价、保质期、原料清单这些属性。
- **Object，对象**，是模板的具体实例。"法棍"和"吐司"是"面包"这个对象类型的两个具体实例。
- **Property，属性**，是对象身上的信息。法棍的"单价=8元"就是一个属性值。
- **Link，链接**，是对象之间的关系。"张三购买了法棍"就是"客户"对象和"面包"对象之间的一条链接。

**在电力行业的应用**

把这个比喻搬到电力行业。一台编号为 SZ-110-001 的 110kV 变压器在 Foundry 里是这样被管理的：

它首先属于"变压器"这个 Object Type。这个类型定义了变压器应该有的属性：额定容量、额定电压、制造商、出厂日期、安装位置、运行状态。SZ-110-001 作为一个具体的 Object，它的属性值是：额定容量 31500kVA，额定电压 110kV，制造商特变电工，出厂日期 2018-06-15，安装位置深圳南山变电站，运行状态正常。

更重要的是，这台变压器和其他对象之间有丰富的 Link 关系。它**安装在**"深圳南山变电站"这个"变电站"对象上。它**属于**"南山供电局"这个"组织"对象。它**关联到**"2018-南山-设备采购合同-003"这个"合同"对象。它最近被"张工-巡检记录-20260301"这个"巡检记录"对象引用。

当一个管理者想知道"这台变压器的采购合同是否已经付款完成"时，他不需要分别打开设备台账系统、合同管理系统和财务系统。在 Foundry 中，他只需要从变压器对象出发，沿着"关联合同"这条 Link 找到合同对象，再沿着"关联付款记录"这条 Link 找到付款状态。这就是 Ontology 的价值：**让数据之间的关系变得可以直接"走"过去，而不是需要在多个系统之间手动查找和比对**。

**Palantir 的 Ontology 是人工定义的**

这一点非常重要，也是 YIMO 差异化的起点。在 Foundry 中，Object Type 需要由管理员手动定义。管理员需要决定：我的组织里有哪些类型的"东西"？每种东西有哪些属性？东西和东西之间有什么关系？这个过程通常需要领域专家参与，耗时数天甚至数周。比如，定义电力行业的 Ontology，需要懂电力业务的人和懂数据建模的人坐在一起，逐一梳理"变压器""线路""变电站""合同""巡检记录"等几十种对象类型及其属性和关系。

### 2.2 Pipeline（数据管道）—— 数据怎么进入 Foundry

Ontology 定义了"数据应该长什么样"，Pipeline 解决的是"数据从哪里来"。

现实中，企业的数据分散在几十个系统中。ERP 系统用 SAP 的数据库，财务系统用 Oracle，设备台账可能还在 Excel 里，现场巡检数据在移动端 App 的 SQLite 中。Foundry 的管道构建器 Pipeline Builder 提供了 100 多个预置的数据连接器，涵盖 SAP、Oracle、MySQL、PostgreSQL、Snowflake 等数据库，也支持 CSV、Excel、JSON 文件导入和 REST API 数据获取，甚至可以接入实时数据流处理平台 Kafka 接收实时事件。

数据进入 Foundry 后，通常需要经过清洗和转换，这在 Foundry 中叫做 Transform（转换）。Transform 使用 Python 或 SQL 编写，定义了数据清洗的规则。举一个具体的例子：南方电网的 ERP 系统中，设备编号格式是 "SZ-110-001"，但 GIS 系统中同一台设备的编号是 "SZNSVARS001"。Transform 可以定义一个规则，把两种编号对齐到统一格式。清洗后的数据被写入 Foundry 的内部存储，形成一个"干净的"数据集。

Pipeline 还支持定时调度。你可以设置 Pipeline 每天凌晨 2 点自动从 ERP 同步最新数据，或者每 5 分钟从 SCADA 系统拉取一次设备运行数据。对于时效性要求极高的场景，还可以配置事件触发，比如设备状态变更时立即同步。

### 2.3 Workshop（工作流编排）—— 用对象做事

有了 Ontology 定义数据长什么样，有了 Pipeline 解决数据从哪里来，下一步就是 Workshop 工作坊，用数据做事情。

Workshop 的核心概念是 Action，即动作。Action 可以理解为"一个可以操作对象的函数"。比如：

- **审批动作**：当一个"合同"对象的金额超过 300 万元时，自动把这个合同的状态改为"待审批"，并通知相关审批人。这个 Action 的输入是合同对象，输出是状态变更和通知。
- **设备调度动作**：当一台变压器的负载率超过 85% 时，自动查找同一变电站中负载率最低的变压器，并生成一条"负载转移建议"。
- **合规检查动作**：每月自动扫描所有"巡检记录"对象，找出超过 30 天未巡检的设备，生成逾期巡检报告。

Workshop 最强大的特性是写回能力 Write-back。大多数数据分析平台是"只读"的，你可以查看数据、分析数据、生成报告，但不能改变源系统中的数据。Foundry 不同，它可以把操作结果写回到源系统中。比如，审批通过的合同不仅在 Foundry 中标记为"已批准"，还可以自动更新 ERP 系统中对应合同的状态字段。这意味着用户可以在 Foundry 中完成端到端的业务流程，而不仅仅是查看数据。

### 2.4 Quiver 与 Slate（可视化）—— 看到全貌

Foundry 提供两套可视化工具：Quiver 和 Slate。

**Quiver** 是一个拖拽式看板构建器，面向不懂编程的业务用户。用户可以把图表、地图、表格、时间线等组件拖到画布上，绑定 Ontology 中的数据，几分钟内就能搭建出一个交互式看板。例如，一个"变电站设备健康概览"看板可以包含：左侧是广东省地图，点亮所有变电站位置；右上是设备健康评分的柱状图；右下是近 30 天故障次数的时间线。所有组件联动，点击地图上某个变电站，右侧的图表自动筛选到该变电站的数据。

**Slate** 面向开发者，提供更大的自由度。开发者可以用 HTML/CSS/JavaScript 自定义界面，调用 Foundry 的 API 获取 Ontology 数据。Slate 适合构建高度定制的业务应用，而非通用看板。

Foundry 还原生支持地理空间可视化，这对电力行业特别重要。线路、变电站、铁塔都有地理坐标，在地图上可视化可以直观展示资产分布、故障范围和巡检路线。

### 2.5 AIP —— 2023 年之后的新方向

2023 年，随着 GPT-4 等大语言模型的出现，Palantir 迅速推出了人工智能平台 AIP，将大语言模型集成到 Foundry 中。AIP 的全称是 Artificial Intelligence Platform。

AIP 的核心理念是"让 LLM 操作 Ontology"。传统做法是：用户学习 Foundry 的界面和查询语法，手动查找和操作对象。AIP 的做法是：用户用自然语言告诉系统想做什么，比如"帮我找出广东省所有运行超过 15 年的变压器，按故障率排序"，AIP 会自动把这句话翻译成对 Ontology 的查询和操作。

AIP 使用检索增强生成技术 RAG。简单来说，LLM 本身不知道你公司的数据，因为它只在公开互联网数据上训练。RAG 的做法是在回答问题之前先从 Ontology 中检索相关信息，把检索结果作为上下文提供给 LLM。这样 LLM 就能基于你公司的真实数据来回答问题，而不是编造答案。

AIP 还支持 LLM 直接触发 Action。比如用户说"把上个月故障超过 3 次的变压器标记为需要检修"，AIP 不仅查找符合条件的变压器，还能自动调用"标记检修"这个 Action，修改对象状态。这种"自然语言→理解意图→操作对象→写回结果"的能力，是 Foundry 2023 年之后最大的变化。

## 3. Palantir 的真实客户案例

抽象的技术描述不如具体的案例。以下四个案例来自 Palantir 的公开资料和客户证言，展示了 Foundry 在不同行业的实际应用。

### 3.1 空客 Airbus —— 飞机制造的数据操作系统

空客是全球最大的飞机制造商之一，总部在法国图卢兹。制造一架 A350 宽体客机涉及超过 300 万个零件，供应链覆盖 30 多个国家的上千家供应商。空客面临的核心问题是：当一个零件出了质量问题，如何在数小时内找到所有使用了这个零件的飞机，并评估影响范围？

在使用 Foundry 之前，回答这个问题需要依次查询四个独立系统：设计系统确认哪些机型用了这个零件，生产系统确认哪些飞机已经装了这个零件，交付系统确认哪些飞机已经交给航空公司，维护系统确认这个零件在运行中是否已经出过问题。每个系统都有自己的零件编码方式，人工追溯需要几天到几周。

空客用 Foundry 建立了一个统一的 Ontology，将"零件""飞机""供应商""生产批次""交付记录""维护记录"等全部建模为对象，并通过 Pipeline 从各个源系统实时同步数据。现在，当一个零件出现质量预警时，供应链团队可以在几分钟内沿着对象关系链追溯：这个零件→来自哪个供应商的哪个批次→装在了哪些飞机上→这些飞机交付给了哪些航空公司→它们最近一次维护检查是什么时候。

### 3.2 BP 石油 —— 资产健康监控

英国石油公司 BP 在全球运营着数千口油井、数百个海上平台和数万公里的管道。每一口油井和每一个平台上都有成百上千的传感器，实时采集温度、压力、流量、振动等数据。BP 的挑战是：如何从每天产生的数十亿条传感器数据中，及早发现设备异常，在故障发生之前进行预防性维护？

BP 用 Foundry 将"油井""泵""阀门""管道"等物理资产建模为对象，每个对象关联其传感器数据流。Pipeline 从 SCADA 系统实时拉取传感器数据，Transform 计算滑动窗口统计量，包括最近 24 小时的平均值、标准差和趋势斜率。当某个传感器的读数偏离历史基线超过 3 个标准差时，系统自动生成预警对象，通知现场运维团队。BP 报告称，使用 Foundry 后非计划停机减少了 50% 以上。

这个案例对电力行业特别有参考价值。南方电网的变压器、断路器、线路同样拥有大量传感器数据，同样需要从设备台账→传感器数据→故障预测→维修工单的端到端管理。

### 3.3 NHS —— 新冠疫苗分配

2020-2021 年新冠疫情期间，英国国家医疗服务体系 NHS 面临一个前所未有的物流挑战：如何将数千万剂疫苗高效分配到 1200 多个接种点，同时满足疫苗的冷链要求（辉瑞疫苗需要 -70°C 存储）、合理安排人群优先级，并匹配接种点容量？

NHS 用 Foundry 将"疫苗批次""接种点""冷链设备""预约记录""人群优先级"建模为对象。Pipeline 从供应商系统实时获取疫苗到货信息，从预约系统获取各接种点的预约量，从冷链监控系统获取存储温度。Workshop 中的 Action 自动执行疫苗分配逻辑：根据各接种点的预约量、冷链容量和距离，计算最优配送方案。当某个接种点的疫苗即将过期时，系统自动触发调拨，将疫苗转移到消耗更快的接种点。

这个案例展示了 Foundry 超越传统"数据分析"的能力：它不仅帮你理解数据，还能直接驱动业务操作。

### 3.4 美国军方 —— 战场情报融合

虽然 Gotham 的细节很少公开，但通过国会听证会和国防部公开报告可以了解到：Gotham 在阿富汗和伊拉克战争中被用于追踪简易爆炸装置网络。它将通信监听数据、无人机侦察影像、线人报告、金融转账记录等多源情报整合到一个统一平台中，帮助分析师发现制造和放置爆炸装置的完整网络：谁制造、谁运输、谁安放、资金从哪来。

2022 年俄乌冲突爆发后，Palantir 向乌克兰军方提供了 Foundry 和 Gotham 的修改版本，用于整合战场态势信息。据公开报道，该系统将卫星图像、开源情报和部队位置信息融合，支持指挥官快速做出战术决策。

### 3.5 总结：为什么大型组织愿意付高昂费用

Palantir 的年度订阅费用通常在数百万到数千万美元之间。为什么客户愿意支付这样的价格？核心原因有三个。

第一，**Foundry 解决的问题是跨系统的**。企业内部任何一个部门的 IT 系统都只能看到自己的数据。Foundry 是极少数能够将跨部门、跨系统的数据统一到一个操作平台上的产品。仅仅是这种"全局视角"的能力，就已经值回了大部分费用。

第二，**Foundry 不仅分析数据，还能操作数据**。Workshop 的 Action 和 Write-back 机制意味着用户可以在 Foundry 中完成端到端的业务流程。它不是一个"看看报表然后去别的系统操作"的工具，而是一个"在这里发现问题、在这里解决问题"的操作平台。

第三，**Palantir 提供驻场工程师**。大型客户通常会有 Palantir 的前沿部署工程师 FDE 团队常驻现场。FDE 深入理解客户业务，帮助定义 Ontology、开发 Pipeline、构建 Workshop 应用。这种"软件+服务"的模式确保了落地效果，但也是高费用的重要原因。

# 第二部分：YIMO 做了什么 — 诚实的能力评估

## 4. YIMO 的核心能力

### 4.1 自动对象抽取 — YIMO 的真正创新点

YIMO 最有价值的能力是**自动对象抽取**，这恰好是 Palantir Foundry 依赖人工完成的环节。

前面介绍了，Foundry 的 Ontology 需要管理员手动定义 Object Types。这个过程通常需要领域专家和数据工程师合作，耗时数天到数周。YIMO 的做法不同：它从南方电网的数据架构文档（DA-01/02/03 Excel 文件）中**自动发现**业务对象。

具体算法分为五个步骤：

**第一步，数据采集**。DataArchitectureReader 模块读取 Excel 文件中的三层架构实体名称。概念实体如"项目管理""设备运维"，逻辑实体如"项目立项申请表""合同审批单"，物理实体如"t_project_info""t_contract_detail"。

**第二步，语义向量化**。每个实体名称被输入 SBERT 模型 shibing624/text2vec-base-chinese，转换为一个 768 维的数字向量。这个向量捕获了名称的语义含义，比如"项目立项"和"工程启动"虽然字面不同，但向量非常接近。SBERT 的全称是 Sentence-BERT，即句子级别的 BERT 模型。BERT 是 Google 在 2018 年发布的一种预训练语言模型，专门为理解自然语言语义而设计。

**第三步，语义聚类**。使用 AgglomerativeClustering（凝聚层次聚类算法）将语义相近的实体自动归为同一组。聚类数量由自适应公式决定：`目标聚类数 = max(5, min(20, 实体总数 ÷ 20))`。比如 356 个实体会被分成大约 15-18 个聚类。每个聚类代表一个潜在的业务对象，聚在一起的实体共享一个抽象概念。

**第四步，对象命名**。每个聚类需要一个有意义的名称。YIMO 支持两种命名方式。规则命名通过分析聚类内实体名称的高频关键词自动提取命名，例如一个聚类包含"项目立项""项目预算""项目进度"等实体时，提取出关键词"项目"作为对象名称。LLM 命名调用 DeepSeek 大语言模型 API，将聚类中的代表性实体名称发给模型，让模型归纳出一个抽象的对象名称。LLM 命名通常质量更高，但需要网络连接。

**第五步，垃圾名称过滤与层级关联构建**。系统自动检测低质量的对象名称，比如纯英文缩写或单字符中文，将其标记为待合并对象。同时，HierarchicalRelationBuilder 模块建立每个对象与三层架构实体之间的关联关系，记录关联类型、关联强度和匹配方法。关联类型分为直接、间接和聚类三种，关联强度为 0-1 的连续值。

**实际效果**：在最新一次抽取中，YIMO 从输配电域的 3 个 Excel 文件中自动抽取了 **8 个对象**，分别是计划、设备、系统、指标、文档、监督、班站和项目，建立了 **12,282 条**对象-实体关联关系。从计划财务域的 3 个 Excel 文件中抽取了 **7 个对象**，分别是指标、资产、合同、票据、发电、监督和项目，建立了 **1,294 条**关联关系。整个过程无需任何人工干预，耗时约 2-3 分钟。

### 4.2 三层架构可视化

YIMO 的可视化系统基于 ECharts 图表库构建，提供多种视角来展示对象与三层架构实体的关系。

**力导向知识图谱**以对象为中心节点，围绕展示其关联的概念实体、逻辑实体和物理实体。三种层级的节点用不同颜色区分：概念层紫色、逻辑层绿色、物理层橙色。用户可以点击展开或折叠节点，查看特定对象的关联详情。

**Sankey 流向图**展示从对象到三层架构的四层数据流向，直观呈现每个对象关联了多少概念实体、逻辑实体和物理实体，以及不同层级之间的穿透关系。

**治理看板**提供 8 项数据治理指标的实时监控，包括三层关联完整性、关联强度分布、跨域对象覆盖率等。治理看板帮助数据管理人员快速定位数据质量问题，比如哪些对象缺少物理层关联、哪些域的数据覆盖不完整。

## 6. 差距的本质：PoC 到产品的鸿沟

在讨论路线图之前，需要理解 YIMO 当前所处的位置。软件产品的成熟度通常分为四个阶段：

1. **概念验证 PoC**：证明核心算法可行，能在受控环境下工作。
2. **最小可用产品 MVP**：核心功能可用，有基本的用户界面和 API，可以做演示。
3. **生产就绪 Production-Ready**：有认证授权、错误处理、监控告警、数据备份、API 分页等企业级特性，可以部署到客户环境。
4. **企业级产品 Enterprise-Grade**：多租户、高可用、合规审计、SDK/文档完善、有技术支持体系。

**YIMO 当前处于 PoC 和 MVP 之间。** 核心算法（对象抽取）已经过了 PoC 阶段，在两个真实数据域上验证通过。REST API 和前端界面将其推向了 MVP 的边缘。但从 MVP 到 Production-Ready 还有大量工程工作需要完成。Palantir Foundry 是一个经过 20 年打磨的 Enterprise-Grade 产品，直接"对标"在当前阶段既不现实也没有必要。

## 7. 从 PoC 到产品的 5 步路线

以下路线图聚焦于将 YIMO 从当前的 PoC/MVP 状态推进到 Production-Ready 状态。每一步都标注了具体需要开发的代码模块和预估代码量，便于评估工作量。

### Step 1：数据质量与抽取优化（预估 2-3 周）

**目标**：确保对象抽取的输出质量达到可以交付给客户的标准。

**具体工作**：

1. **跨域对象去重算法**。当前两个域分别抽取后可能产生语义相同但名称不同的对象，例如输配电域的"设备"和计财域的"资产"可能指同一概念。需要开发基于 SBERT 余弦相似度的跨域去重模块，阈值可配置，决策记录可审计。这部分代码已有表结构 `object_dedup_decisions` 和 API 端点 `/api/olm/cross-domain-duplicates`，但缺少去重决策的执行逻辑和用户确认流程。预估新增约 300 行 Python 代码。

2. **抽取质量评估指标**。为每次抽取结果计算质量评分，包括三个维度：聚类内聚度衡量聚类内实体的平均语义相似度，聚类分离度衡量不同聚类之间的平均语义距离，对象命名质量则检查名称是否出现在领域术语库中。预估新增约 200 行代码。

3. **增量抽取模式**。当前每次抽取都是全量重新运行。当客户添加新的 Excel 文件时，应该支持增量抽取，只处理新增数据，与已有对象进行对比和合并。预估新增约 500 行代码。

### Step 2：安全与认证（预估 4-6 周）

**目标**：所有 API 端点受认证保护，支持基于角色的访问控制。

**具体工作**：

1. **JWT Token 认证中间件**。使用 Flask-JWT-Extended 扩展，实现用户注册、登录、Token 刷新。所有 API 端点添加 `@jwt_required()` 装饰器。需要新增 `users` 表（用户名、密码哈希、角色、创建时间）和 `refresh_tokens` 表。预估新增约 400 行 Python 代码 + 100 行 SQL。

2. **RBAC 角色权限模型**。定义四个角色：admin 管理全部资源，architect 执行抽取和管理对象，analyst 查看和使用分析功能，viewer 只读浏览。使用装饰器模式实现权限检查，如 `@role_required('architect')`。预估新增约 200 行代码。

3. **审计日志表**。新增 `audit_logs` 表，记录每个 API 调用的操作人、操作时间、端点、请求参数、响应状态码。通过 Flask 的 `after_request` 钩子自动记录。预估新增约 200 行 Python + 50 行 SQL。

4. **用户管理前端页面**。在 v10.0 界面中新增用户管理面板：用户列表、创建用户、分配角色、登录/登出界面。预估新增约 500 行 HTML/JS。

### Step 3：数据连接器框架（预估 6-8 周）

**目标**：支持从 MySQL、PostgreSQL、REST API 等数据源导入数据，替代当前仅有的 Excel 文件导入。

**具体工作**：

1. **连接器抽象层**。定义 `BaseConnector` 基类，包含 `connect()`、`fetch_schema()`、`fetch_data()`、`test_connection()` 四个抽象方法。每种数据源实现一个子类：`ExcelConnector`（已有，需重构）、`MySQLConnector`、`PostgreSQLConnector`、`RESTAPIConnector`、`CSVConnector`。预估新增约 800 行代码。

2. **连接器配置管理**。新增 `data_connectors` 表，存储每个连接器的类型、连接参数（加密存储）、同步频率。前端新增连接器管理面板，支持创建、测试、编辑和删除连接器。预估新增约 400 行 Python + 300 行 HTML/JS。

3. **增量同步引擎**。基于连接器定期从源系统拉取新数据，与已有数据对比，只导入变化部分。需要在 EAV 表中新增 `last_sync_at` 和 `sync_hash` 字段跟踪同步状态。预估新增约 500 行代码。

### Step 4：工作流引擎升级（预估 8-10 周）

**目标**：将当前 3 个预置机理函数升级为完整的 Action 框架，支持事件驱动的自动化规则。

**具体工作**：

1. **Action 框架**。定义 Action 的标准接口，包含四个方法：`trigger` 负责触发条件判断，`evaluate` 负责评估逻辑，`execute` 负责执行动作，`rollback` 负责回滚。当前的 THRESHOLD/FORMULA/RULE 三种机理函数重构为 Action 的三种子类型。预估新增约 600 行代码。

2. **事件驱动引擎**。当对象属性变更时，自动检查是否有 Action 的触发条件被满足，如果是则执行 Action。需要实现事件总线（Event Bus）模式，新增 `event_queue` 表和后台工作进程。预估新增约 800 行代码。

3. **可视化规则编辑器**。在前端提供拖拽式界面，业务人员无需编写 JSON 即可定义触发条件和执行动作。预估新增约 1000 行 HTML/JS。

### Step 5：生产化部署（预估 4-6 周）

**目标**：使系统可以安全、稳定地部署在客户的生产环境中。

**具体工作**：

1. **API 工程化**。为所有列表 API 添加分页参数 `page` 和 `page_size`，引入速率限制中间件 Flask-Limiter，添加 API 版本前缀 `/api/v1/`。预估修改约 400 行现有代码 + 新增 200 行。

2. **生产级 Web 服务器**。将 Flask 内置开发服务器替换为 Gunicorn，配置 Nginx 反向代理和 SSL 终止。提供 `gunicorn.conf.py` 配置文件和 `nginx.conf` 配置。预估新增约 200 行配置。

3. **健康检查与监控**。扩展现有 `/health` 端点，增加数据库连接池状态、SBERT 模型加载状态、磁盘使用率、最近抽取时间等指标。添加 Prometheus 指标端点供监控系统采集。预估新增约 300 行代码。

4. **CI/CD 流水线**。添加 GitHub Actions 工作流，在每次 push 时自动运行测试套件、检查代码风格、构建 Docker 镜像。预估新增约 200 行 YAML 配置。

5. **数据库迁移工具**。引入 Flask-Migrate（基于 Alembic），管理数据库 Schema 的版本演化。当前的 `bootstrap.sql` 只能创建全新数据库，不支持平滑升级。预估新增约 300 行代码。

## 8. 对标 Palantir 的差异化策略

### 8.1 不要试图复制 Palantir

Palantir 有 20 年的技术积累、数千名工程师、数十亿美元的研发投入。试图用一个开源项目在所有维度上"对标"Palantir 既不可能也没有必要。更务实的策略是：**在一个垂直领域做到 Palantir 做不到的事情。**

### 8.2 YIMO 的三个独特优势

**优势一：自动对象抽取。** 这是 YIMO 相对于 Foundry 的结构性优势。Foundry 的 Ontology 建模依赖人工，YIMO 的对象抽取是自动化的。对于电力行业这种数据架构文档已经标准化（DA-01/02/03）的领域，自动抽取可以将对象建模从"人工数周"压缩到"机器数分钟"。这个优势在 Palantir 不大可能消失，因为 Palantir 面向的是通用市场，没有动力为某一个行业的文档格式做定制化的自动抽取。

**优势二：电力行业深度适配。** YIMO 原生理解南方电网的三层架构标准，即 DA-01 概念实体、DA-02 逻辑实体和 DA-03 物理实体，内置电力行业术语库和常见对象模板。Foundry 是行业无关的通用平台，要在电力行业使用需要大量定制开发。YIMO 的行业深度是 Palantir 的广度所无法覆盖的。

**优势三：内网部署零依赖。** 电力行业的生产系统通常部署在严格隔离的内网中。YIMO 通过三级回退策略确保在完全断网环境下仍可运行：SBERT+LLM 完整模式、SBERT+规则半离线模式、纯规则全离线模式。Docker Compose 一键启动，无需专业运维团队。Foundry 的 Private Cloud 部署需要 Palantir 工程师团队现场支持。

### 8.3 目标定位

YIMO 的目标定位不是"通用数据操作系统"，那是 Palantir 的地盘。YIMO 的定位是：**面向中国电力行业的垂直对象管理平台**。

具体来说：
- **目标客户**：南方电网及其下属供电局、省级电力公司
- **核心场景**：数据架构治理、三层架构关联分析、跨域对象统一
