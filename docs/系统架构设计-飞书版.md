# YIMO 系统架构设计方案

> 版本: v1.0 | 日期: 2026-03-15 | 适用场景: 内部技术汇报

---

## 1. 系统概述

YIMO 是面向南方电网的对象抽取与三层架构关联系统，从企业数据架构文档（DA-01/02/03 Excel）中自动提炼业务对象，建立概念→逻辑→物理三层穿透式关联。

### 解决的核心问题

| 痛点 | 传统方式 | YIMO 方案 |
|------|----------|-----------|
| 对象定义依赖人工 | 领域专家手动梳理，耗时数天/数周 | SBERT 语义聚类 + LLM 归纳命名，分钟级自动抽取 |
| 三层架构关联缺失 | 概念/逻辑/物理实体各自独立，无法穿透 | 自动构建对象→三层实体关联，支持知识图谱可视化 |
| 多域数据不统一 | 输配电、计划财务等域各自为政 | 统一算法跨域抽取，支持跨域重复检测与对象合并 |

### 核心能力一览

| 能力 | 说明 | 代码量 |
|------|------|--------|
| 对象自动抽取 | SBERT 768 维语义聚类 + DeepSeek LLM 归纳命名 | 2369 行 |
| 三层知识图谱 | ECharts 力导向图，对象→概念→逻辑→物理四层穿透 | 前端 3304 行 |
| Sankey 流向图 | 对象→概念→逻辑实体的数据流向展示 | — |
| 穿透式溯源 | 财务结算→项目立项→采购合同→现场施工全链路追踪 | 4 个 API 端点 |
| 机理函数 | 阈值/公式/规则三类业务规则，含表达式求值引擎 | 6 个 API 端点 |
| 风险预警 | 基于机理函数自动触发，多级别预警管理 | 4 个 API 端点 |
| 数据治理看板 | 完整性、缺陷率、跨域对比等 8 项治理指标 | 4 个 API 端点 |

## 2. 总体架构

系统采用四层架构，从下到上依次为：数据源层、处理引擎层、服务层、展示层。

展示层 (Presentation)
- v10.0 仪表盘 / 知识图谱 / Sankey 图 / 治理看板 / 预警面板
- 技术: HTML5/Jinja2 + ECharts 5.0 + CSS3
- 代码量: 3304 行 (10.0.html) + 767 行 (extraction.html)

服务层 (Service)
- Flask 3.0 应用 / 44 个 REST API 端点 / DeepSeek 代理
- 策略: 数据库优先 + JSON 回退（无 DB 亦可运行）
- 代码量: app.py 357 行 + olm_api.py 3311 行

处理引擎层 (Processing)
- SBERT 语义向量化 / 层次聚类 / LLM 命名 / 垃圾过滤 / 去重
- 技术: text2vec-base-chinese 768维 + scikit-learn
- 代码量: object_extractor.py 2369 行 + simple_extractor.py

数据层 (Data Storage)
- MySQL 8.0 (19 表 + 4 视图) / FAISS 向量索引 / JSON 文件
- EAV 模型 / 对象关联表 / 语义指纹表
- 端口: 3307 (避免 WSL2 冲突)

数据源层 (Data Source)
- DATA/shupeidian/ (输配电, 3 Excel, ~26MB)
- DATA/jicai/ (计划财务, 3 Excel, ~11MB)
- 每个 Excel 含 DA-01/DA-02/DA-03 标准化工作表

### 各层组件详情

| 层级 | 组件 | 技术 | 代码量 | 职责 |
|------|------|------|--------|------|
| 展示层 | 主仪表盘 | HTML5/ECharts | 3304 行 | 12 个功能面板，30+ API 调用 |

## 3. 核心算法设计

### 对象抽取五步流程

1. Excel 文件 (DA-01/02/03) → 数据采集 (DataArchitectureReader)
2. → SBERT 向量化 (768 维向量, text2vec-base-chinese)
3. → 语义聚类 (自适应聚类, AgglomerativeClustering)
4. → 命名 (LLM 归纳命名 / 关键词规则命名，二选一)
5. → 去重 + 关联构建 (垃圾过滤, 小对象合并) → 输出

各步骤说明：

| 步骤 | 模块 | 输入 | 输出 | 说明 |
|------|------|------|------|------|
| ① 数据采集 | DataArchitectureReader | Excel 文件 | 实体名称列表 | 读取 DA-01/02/03 三类工作表 |
| ② SBERT 向量化 | SentenceTransformer | 实体名称 | 768 维向量 | 模型: text2vec-base-chinese |
| ③ 语义聚类 | AgglomerativeClustering | 向量矩阵 | 聚类标签 | 余弦距离，自适应聚类数 |
| ④ 对象命名 | LLMObjectNamer / 规则 | 聚类成员 | 对象名称 | DeepSeek API 或关键词提取 |
| ⑤ 去重+关联 | HierarchicalRelationBuilder | 命名聚类 | 对象+关联 | 垃圾过滤+小对象合并+三层关联 |

### 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| TARGET_CLUSTER_COUNT | 自适应 | max(5, min(20, 实体数 ÷ 20)) |
| SMALL_OBJECT_THRESHOLD | 3 | 少于 3 个实体的对象强制合并 |
| SBERT_MODEL_NAME | text2vec-base-chinese | 768 维中文语义模型 |
| REQUIRED_OBJECTS | ["项目"] | 必须存在的对象（甲方要求） |
| 关联强度 | 0.9 / 0.7 | 样本实体 0.9，其他成员 0.7 |

### 三级回退机制

系统设计了三级回退，确保在不同部署环境下均可运行：

| 级别 | 条件 | 命名方式 | 抽取质量 |
|------|------|----------|----------|
| Level 1 | SBERT + DeepSeek API Key | LLM 归纳命名 | 最优 |
| Level 2 | SBERT + 无 API Key | 关键词规则命名 | 良好 |
| Level 3 | 无 SBERT | simple_extractor.py 纯规则抽取 | 基础可用 |

### 实际抽取成果

| 数据域 | 对象数 | 关联数 | 数据量 | 耗时 |
|--------|--------|--------|--------|------|
| 输配电 (shupeidian) | 8 个 | 12,282 条 | 6.7 MB | ~2 分钟 |
| 计划财务 (jicai) | 7 个 | 1,294 条 | 734 KB | ~1 分钟 |

## 5. API 架构

### 总览：44 个 REST 端点，分 6 大类

| 类别 | 端点数 | 代表性接口 | 说明 |
|------|--------|-----------|------|
| 对象管理 | 8 | GET/POST/PUT/DELETE /api/olm/objects | CRUD + 导出 + 合并 |
| 关联与架构 | 5 | GET /api/olm/object-relations/<code> | 三层关联查询 |
| 可视化 | 5 | GET /api/olm/graph-data-three-tier | 知识图谱 + Sankey |
| 分析统计 | 10 | GET /api/olm/cross-domain-duplicates | 跨域检测 + 颗粒度分析 |
| 生命周期与溯源 | 8 | GET/POST /api/olm/traceability-chains | 生命周期 + 溯源链 |
| 治理与预警 | 8 | POST /api/olm/alerts/run-check | 机理函数 + 预警 |

### 数据获取策略：DB 优先 + JSON 回退

1. API 请求 → 尝试 MySQL 查询
2. 成功 → 返回数据
3. 失败 → 读取 outputs/extraction_*.json → 返回数据

设计理由：允许系统在无数据库环境下（如快速演示、甲方现场）依然正常运行。所有 44 个端点均实现此策略。

## 7. 部署架构

### Docker Compose 组件

| 服务 | 镜像 | 端口 | 职责 |
|------|------|------|------|
| mysql | MySQL 8.0 | 3307 | EAV + 对象存储 |

### start.sh 一键启动流程

1. 检测 MySQL (3307) → 未运行则启动 → 等待就绪 (30s)
2. 显示数据库摘要（对象数/关联数/EAV 实体数）
3. 检测 Python 环境（venv → conda → system）
4. 后台启动 Flask → PID 写入 webapp.pid → 日志写入 webapp.log
5. 健康检查 curl /health（等待 15s）
6. 输出访问地址 http://127.0.0.1:5000

## 8. 关键设计决策

| 决策 | 理由 | 收益 |
|------|------|------|
| EAV 模型（而非固定关系表） | 三层架构实体属性异构，无法预定义统一 schema | 新增域/属性无需改表，灵活扩展 |
| DB + JSON 双轨回退 | 甲方现场可能无 MySQL 环境 | 随时可演示，零依赖 |
| 端口 3307 | WSL2 下 3306 被系统占用 | 避免开发环境冲突 |
| 自适应聚类数 | 不同域实体数量差异大 | 自动适配，无需手动调参 |
| LLM 自动检测 | 部署环境可能无外网 | 有 API Key 自动启用，无则规则回退 |
| SBERT 中文专用模型 | 通用英文模型对中文语义理解差 | text2vec-base-chinese 对电力术语效果好 |
| Blueprint 模块化 API | 44 个端点放 app.py 太臃肿 | olm_api.py 独立维护，职责分离 |
| REQUIRED_OBJECTS 保障 | 甲方要求"项目"必须被抽取 | 算法保底机制，确保不遗漏 |
